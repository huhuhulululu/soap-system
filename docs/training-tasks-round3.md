# 四代理训练任务（Round 3）

## 目标

- 在并行任务下保持规则一致性和产出稳定性。
- 训练四角色处理“同轮多需求 + 冲突仲裁 + 审计抽检”。

## 本轮重点机制

1. 同一轮运行 3 个并行小需求（A/B/C）。
2. 若规则冲突，必须走仲裁单，不允许口头决策。
3. 审计员执行抽检，不只看最终汇报。

## 12 个任务（Round 3）

1. 规则掌握者：为 A/B/C 三个需求标注适用规则集与禁改项。
2. 编写者：并行实现 A/B/C，要求每个需求最小改动、独立提交。
3. 检查者：分别对 A/B/C 输出违规清单（不得合并成一份）。
4. 审计员：抽检 30% 规则项，核对检查者是否漏检。
5. 规则掌握者：处理 1 个规则冲突案例并提交仲裁结论。
6. 编写者：根据仲裁结论重构冲突改动（仅改受影响范围）。
7. 检查者：复检冲突修复，确认无新回归。
8. 审计员：检查“冲突处理前后”的证据链完整性。
9. 编写者：完成一次“快速回归修复”（30 分钟内）并记录时间线。
10. 检查者：验证快速修复未破坏单调性/SOA/SOAP 完整性。
11. 审计员：执行发布前抽检（规则、测试、文档三线）。
12. 审计员：发布本轮最终裁决 + 下轮风险清单。

## 规则冲突仲裁单（模板）

```text
[conflict]
conflict_id: CF-YYYYMMDD-01
rule_version: R1.1
conflicting_rules: [X-001, A-00X]
scenario: <冲突场景>
options:
- option: A
  impact: <影响>
- option: B
  impact: <影响>
decision: A|B
rationale: <裁决理由>
owner: Rule Master
auditor_ack: yes|no
```

## 本轮放行条件

- A/B/C 三需求均有独立证据链。
- 冲突项均有仲裁单且被执行。
- 抽检漏检率 <= 5%。
- 回归全绿：
  - `cd frontend && npm test`
  - `cd frontend && BULK_SEED_COUNT=100 npx vitest run engine-random`
